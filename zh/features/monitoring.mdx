Firecrawl 监控可检测网站内容何时发生变化，并通过 Webhook 或电子邮件接收通知。它会定期执行抓取、爬取或网页搜索，并将每次结果与上一次保留的快照进行比较。你可以使用监控来跟踪产品页面、文档、博客、更新日志、竞争对手网站，或任何变化值得关注的页面——也可以监控整个网页，以发现与某个目标匹配的新结果 (请参见 [Web monitoring](#web-monitoring)) 。

每次检查都会将页面级结果记录为 `same`、`new`、`changed`、`removed` 或 `error`。你可以在每个受监控页面处理完成时接收 Webhook，也可以在每次检查完成后接收一个 Webhook；还可以在发生更改或错误时接收电子邮件摘要，或组合使用这些通知方式。

<div id="goals-and-judging">
  ## 目标与判定
</div>

如果你只想在发生有意义的变化时收到告警，请添加一个通俗易懂的 `goal`。如果设置了 `goal` 但省略了 `judgeEnabled`，Firecrawl 会自动启用判定。判定会在已发生变化的页面上运行，并返回一个 `judgment`，其中包含 `meaningful`、`confidence`、`reason` 和 `meaningfulChanges`。

如果你想先保存目标、暂时不对变化进行判定，请使用 `judgeEnabled: false`。只有当监控同时设置了 `judgeEnabled` 且 `goal` 非空时，判定器才会运行。

<Note>
  每次检查都会对底层抓取或爬取照常计费。如果启用了判定，判定器还会为其验证的每个已发生变化的页面额外收取 1 个额度。没有已发生变化页面的检查不会消耗判定额度。
</Note>

好的目标应简短且明确：说明什么情况应触发告警，重申任何范围限制，例如前 N 项、价格、角色类型、公司、地区、主题、状态或实体，并且仅在排除条件本身属于目标意图时才写入。如果目标范围较宽泛，就保持宽泛；例如，&quot;任何变化&quot; 不应再加入会掩盖变化的噪声过滤条件。

例如，一个具有以下目标的监控：

```text
当与 AI 相关的新 Hacker News 故事进入前 10 时发出告警。忽略与 AI 无关的故事的变动。不对前 10 以外的变动发出告警。
```

<div id="targets">
  ## 目标
</div>

监控支持三种目标类型：

* `scrape`：对 `urls` 中的每个 URL 执行一次抓取。
* `crawl`：每次检查时，对 `url` 执行一次完整爬取，然后对所有发现的页面进行差异比对。
* `search`：每次检查时对 `queries` 执行网页搜索，并在出现新的匹配结果时发出告警。其字段和行为请参见 [网页监控](#web-monitoring)。

每个监控可接受 1 到 50 个目标。`retentionDays` 的默认值为 `30`，最大可设置为 `365`。

目标的抓取选项会传递到底层抓取任务。由监控触发的抓取默认将 `maxAge` 设为 `0`，因此除非你显式设置其他 `maxAge`，否则每次检查都会执行一次新的抓取。

<div id="web-monitoring">
  ## 网页监控
</div>

抓取监控和爬取监控会监控你指定的 URL。**网页**监控则监控整个网络。它不是对你已知的页面做差异比对，而是由你提供要运行的 `queries` 和一个 `goal`，然后 Firecrawl 在每次检查时执行这些查询，并针对判定器认为与目标相关的**新**结果发出告警。它侧重于发现，而不是比对差异。

每次检查都会经历相同的流程：获取目标及其 `queries`，应用时效窗口，运行搜索，按 canonical URL 对结果去重，再由可选的 AI 判定器判断哪些新结果对该目标有意义，并通过与抓取监控和爬取监控相同的 webhook 和电子邮件渠道发出告警。调度、目标、判定和通知的工作方式都与上文所述完全一致。

<Note>
  页面监控和爬取监控会对你指定的 URL 内容做差异比对；网页监控则是在整个网络中发现新结果。底层的调度、判定器和通知机制都相同。
</Note>

<div id="search-target">
  ### 搜索目标
</div>

搜索目标使用 `type: "search"`，并将 `urls` 替换为要执行的查询及结果评分方式：

```json Search target
{
  "type": "search",
  "queries": ["open source AI coding assistant launch"],
  "searchWindow": "24h",
  "maxResults": 10,
  "includeDomains": ["news.ycombinator.com"]
}
```

| 字段               | 类型                                               | 描述                                                                                                                 |
| ---------------- | ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------ |
| `type`           | `"search"`                                       | 选择搜索目标。                                                                                                            |
| `queries`        | `string[]`                                       | 每次检查时执行的搜索查询。                                                                                                      |
| `searchWindow`   | `"5m" \| "15m" \| "1h" \| "6h" \| "24h" \| "7d"` | 时效过滤器——仅考虑在该时间窗口内发布的结果。                                                                                            |
| `maxResults`     | `number`                                         | 每次检查要评估的结果总数，`1`–`50`。默认值为 `10`。这是所有 `queries` 共享的总上限 (结果会先合并并去重) ，而不是单个查询的上限——因此，如果其他查询先占满上限，某个查询贡献的结果可能会更少，甚至为零。 |
| `includeDomains` | `string[]`                                       | 可选。将结果限制在这些域名内。与 `excludeDomains` 互斥。                                                                              |
| `excludeDomains` | `string[]`                                       | 可选。排除来自这些域名的结果。与 `includeDomains` 互斥。                                                                              |

<Note>
  **额度会随查询数量增加而增长。** 每个查询最多获取 `maxResults` 条结果，搜索调用的额度按所有查询在*合并和去重之前*的原始结果计费——因此，增加查询会提高实际搜索成本，而不仅仅是前期估算值。合并/去重后，Firecrawl 最多评估 `maxResults` 个选中的候选结果，因此判定额度仍会受所选/判定结果数量上限的约束。
</Note>

监控级别的 `goal` 和 `judgeEnabled` 的作用与 [目标与判定](#goals-and-judging) 中相同。`queries` 为必填；`goal` 是判定器用来对每个新结果评分的依据，不会用于生成查询。

<div id="writing-good-goals-and-queries">
  ### 编写好的 goals 和 queries
</div>

网页监控的质量取决于两个各自发挥不同作用的关键因素：

* **`queries` 控制召回率** —— 决定每次搜索能找回什么内容。太窄，真实事件就永远不会出现；太宽，judge 就得花额度去过滤噪声。
* **`goal` 控制精确率** —— 决定哪些检索结果会真正触发告警。judge 会根据 goal 为每条新结果打分，因此 goal 才是区分“真正匹配”和“虽然相关但并不重要”的关键。

两者都需要调优。再完美的 goal，也无法对 queries 从未检索到的结果触发告警；而宽泛的 queries 配上模糊的 goal，则会持续产生低价值告警。

**好的 queries 应该写得像搜索词，而不是完整句子：**

* 用关键词，不要写成自然语言句子 —— 用 `OpenAI new model release`，不要用 `tell me when OpenAI releases a new model`。
* 多词实体要加引号 (`\"Llama 4\"`) ，同义词用 `OR` 分组 (`launch OR release OR announcement`) 。
* 每个 query 都要尽量简洁 (约 2–6 个词) 。一个宽泛的 query 通常比多个狭窄的 query 更好 —— 额外的 queries 只会分摊 `maxResults` 预算，却不会增加覆盖面。
* 每个*不同*主题只用一个 query。一个主题即使有多个侧面 (“发布、基准测试、文档”) ，也仍然是一个 query；只有当 goal 明确点出彼此独立的实体时才需要拆分 (例如 “OpenAI、Anthropic 和 Google”) 。
* 不要在 queries 中使用 `site:` / `-site:` 操作符 —— 改用 `includeDomains` / `excludeDomains`。

**好的 goals 要用直白的语言说明匹配条件，并且只在排除项本身属于意图的一部分时才添加排除：**

* 说明主题，以及什么才算匹配：*&quot;当 OpenAI 宣布全新模型时告警。&quot;*
* 消除有歧义的术语：*&quot;Firecrawl (网页抓取 API) &quot;* 可以避免 judge 跑去匹配消防设备。
* 只有在确实属于意图范围的不匹配项上才添加 `Ignore ...`：*&quot;忽略观点文章、教程和无关的 AI 新闻。&quot;* 不要重复描述通用噪声 —— judge 已经会处理格式、跟踪参数和重新索引带来的波动。
* 如果意图本身很宽泛，就保持宽泛。过度收紧的 goals 会压掉真实匹配。

**健康的监控应该是什么样。** 一个调优良好的网页监控大多数时候都不会报告任何内容 —— 大多数 check 都会返回 `new: 0` —— 只有在真正出现新的、且符合 goal 的内容时才会告警。你可以通过查看 check 摘要以及每条结果的 `searchStatus` (请参见 [Statuses and dedup](#statuses-and-dedup)) 来判断它是否调优得当：

* 持续出现 `ignored` 结果，说明 queries 拉回了噪声，随后又被 goal 拒绝 —— 收紧 queries (或缩小 `searchWindow`) ，避免继续为那些永远不会触发告警的结果支付 judge 成本。
* 频繁出现 `watching`，说明 goal 有歧义 —— 进一步明确匹配标准，让 judge 能做出判断。
* 活跃主题长时间没有结果，说明 queries 太窄，或者 `searchWindow` 太紧 —— 放宽关键词或扩大时间窗口。
* 用户会忽略的告警，说明 goal 过宽 —— 添加与意图相关的 `Ignore ...`。

理想结果是：在保证足够召回率的同时实现高精确率；每一条告警都值得采取行动，同时不会漏掉任何真实事件。

<div id="judging">
  ### 判定
</div>

每次检查对每条结果执行多少处理，由监控的 `judgeEnabled` 控制——也就是[目标与判定](#goals-and-judging)中介绍的同一个标志。启用判定后，Firecrawl 会抓取每个匹配结果，并根据目标评估其内容；除搜索调用外，每条经过判定的结果还会额外计费 1 个额度。设置 `judgeEnabled: false` 时，网页监控会返回去重后的搜索结果，不进行任何 AI 判定——只返回新的 SERP 命中，并且只消耗搜索调用的额度 (每 10 条结果 2 个额度) 。

<div id="statuses-and-dedup">
  ### 状态与去重
</div>

搜索结果使用与抓取监控和爬取监控**相同的页面级 `status` 枚举**，因此现有的 webhook 和检查结果消费者无需改动即可继续工作。搜索结果会映射为：

* `new` — 首次匹配 `goal` 的结果。这种结果会触发告警。
* `same` — 在之前某次检查中已经见过的结果 (不会触发新的告警) 。
* `error` — 无法评估的结果 (例如，用于判定的抓取被跳过了) 。

更细粒度的搜索处理状态会显示在每个页面的 `metadata.searchStatus` 中，取值之一为：

| `searchStatus` | Page `status` | Meaning                        |
| -------------- | ------------- | ------------------------------ |
| `alert`        | `new`         | 判定为有意义的新结果——会触发通知。             |
| `already_seen` | `same`        | 指纹与之前检查中的某个结果匹配。               |
| `watching`     | `same`        | 判定器暂时还不够确定的新结果；会被跟踪，但不会告警。     |
| `ignored`      | `same`        | 判定器认为对 `goal` 没有意义的新结果。        |
| `skipped`      | `error`       | 本次检查中该结果无法判定 (例如抓取失败或判定能力降级) 。 |

结果在首次以 `new` 出现时只会告警一次。去重仅以 canonical URL 为键 (title 和 snippet 会被刻意排除在指纹之外，因此 title/snippet 的变化不会再次触发) 。由于键是 URL，同一现实事件如果出现在多个文章 URL 中，会按每个 URL 分别告警一次，而不是按每个事件告警一次。

编辑监控的 `goal` 或 `queries` 会提升其 `goalVersion`，从而使之前的判定结果失效。重新评估采用惰性方式，而不是批量重新判定：现有结果不会一次性全部重新判定。相反，每个结果会在下一次检查中再次出现时才重新判定，并在那时应用新的 `goalVersion`。未再次出现的结果会一直保留旧的判定结果和 `goalVersion`，直到重新出现。

<div id="create-a-web-monitor">
  ### 创建网页监控
</div>

网页监控的创建方式与抓取监控相同，唯一的区别是目标配置 (`type: "search"`，并使用 `queries`、`searchWindow`、`maxResults` 以及可选的域名过滤器) ，且不需要 URL：

<CodeGroup>
  ```python Python
  from firecrawl import Firecrawl

  firecrawl = Firecrawl(
    # 监控端点需要 API 密钥：
    api_key="fc-YOUR-API-KEY",
  )

  monitor = firecrawl.create_monitor(
      name="AI coding assistant launches",
      schedule={"text": "every 30 minutes", "timezone": "UTC"},
      goal="Alert when a new open-source AI coding assistant is announced. Ignore funding rounds and unrelated AI news.",
      targets=[
          {
              "type": "search",
              "queries": ["open source AI coding assistant launch"],
              "searchWindow": "24h",
              "maxResults": 10,
          }
      ],
      notification={
          "email": {
              "enabled": True,
              "recipients": ["alerts@example.com"],
              "includeDiffs": True,
          }
      },
  )

  print(monitor.id)
  ```

<div id="pricing">
  ## 定价
</div>

监控不单独按每个监控收费。每次检查都按其执行的底层 抓取、爬取 或 搜索 计费；此外，如果启用了有效变更判定，则每个发生变更且经判定器确认的页面还会额外消耗 1 个额度。

| 组件                              | 额度                                  |
| ------------------------------- | ----------------------------------- |
| 抓取 监控                           | 每次检查每个 URL 1 个额度                    |
| 爬取 监控                           | 每次检查每个发现的页面 1 个额度                   |
| 网页 监控                           | 每次检查每 10 个结果 2 个额度                  |
| 网页 监控判定                         | 启用 AI 判定时，每个判定结果 1 个额度 (包含抓取和评估结果)  |
| 启用有效变更判定                        | 每个经判定器确认的变更页面额外 1 个额度               |
| 格式附加项 (JSON、PDF、question、增强模式等) | 与单独使用 [抓取](/zh/features/scrape) 相同     |

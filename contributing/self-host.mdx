Self-hosting Firecrawl lets you run the entire scraping platform on your own infrastructure. This is ideal for organizations with strict security requirements, custom configuration needs, or those looking to contribute to the project.

- **Security and compliance:** Keep all data handling within your controlled environment. Firecrawl is a Mendable product with SOC2 Type2 certification, and self-hosting gives you additional control over data residency.
- **Custom services:** Tailor services like the Playwright service to handle specific use cases not covered by the standard cloud offering.
- **Community contribution:** Gain a deeper understanding of the platform and make meaningful contributions to the project.

If you have questions or need help getting started, join the [Discord community](https://discord.gg/gSmWdAkdwd) or [open a GitHub issue](https://github.com/firecrawl/firecrawl/issues/new/choose).

## Cloud vs. self-hosting

| Capability | Cloud | Self-hosting |
| --- | --- | --- |
| All API endpoints supported | Yes | Not always; `/agent` and `/browser` are not supported |
| Screenshot support | Yes | Yes, when the Playwright service is running |
| Local LLMs (e.g., Ollama) | Not supported | Supported via `OLLAMA_BASE_URL` (experimental) |
| Fire-engine (IP blocks, bot detection) | Yes | Not available |

<Note>
Self-hosted instances do not have access to Fire-engine, which provides advanced features for handling IP blocks and robot detection. Basic scraping with fetch and Playwright is fully supported, but more complex scenarios may require additional proxy configuration.
</Note>

## Prerequisites

Install [Docker](https://docs.docker.com/get-docker/) before proceeding.

## Getting started

### 1. Set environment variables

Create a `.env` file in the root directory. You can copy the template from `apps/api/.env.example`.

The following minimal configuration starts Firecrawl without authentication or optional sub-services (PDF parsing, JS blocking, AI features):

```bash .env
# ===== Required ENVS ======
PORT=3002
HOST=0.0.0.0

# Note: PORT is used by both the main API server and worker liveness check endpoint

# To turn on DB authentication, you need to set up Supabase.
USE_DB_AUTHENTICATION=false
```

### 2. Build and run

```bash
docker compose build
docker compose up
```

Firecrawl is now running at `http://localhost:3002`.

You can access the Bull Queue Manager UI at `http://localhost:3002/admin/{BULL_AUTH_KEY}/queues` (the default key is `CHANGEME`).

### 3. Test the API

Verify your instance is working by sending a crawl request:

```bash
curl -X POST http://localhost:3002/v2/crawl \
    -H 'Content-Type: application/json' \
    -d '{
      "url": "https://docs.firecrawl.dev"
    }'
```

## Environment variables

The table below lists all available environment variables. Only `PORT`, `HOST`, and `USE_DB_AUTHENTICATION` are required.

| Variable | Description | Default |
| --- | --- | --- |
| `PORT` | Port for the API server and worker liveness check | `3002` |
| `HOST` | Host address to bind the server | `0.0.0.0` |
| `USE_DB_AUTHENTICATION` | Enable Supabase-based authentication | `false` |
| `BULL_AUTH_KEY` | Key to access the Bull Queue admin panel | `CHANGEME` |

### AI features

AI-powered features (JSON format on scrape, `/extract` API, summary format, branding format, change tracking format) require an LLM provider.

| Variable | Description |
| --- | --- |
| `OPENAI_API_KEY` | OpenAI API key for AI features |
| `OPENAI_BASE_URL` | Custom OpenAI-compatible API base URL |
| `OLLAMA_BASE_URL` | Ollama server URL (experimental), e.g. `http://localhost:11434/api` |
| `MODEL_NAME` | Model name for Ollama, e.g. `deepseek-r1:7b` |
| `MODEL_EMBEDDING_NAME` | Embedding model name for Ollama, e.g. `nomic-embed-text` |

### Proxy

| Variable | Description |
| --- | --- |
| `PROXY_SERVER` | Full URL (e.g. `http://0.1.2.3:1234`) or IP:port combo (e.g. `0.1.2.3:1234`) |
| `PROXY_USERNAME` | Proxy username (omit if unauthenticated) |
| `PROXY_PASSWORD` | Proxy password (omit if unauthenticated) |

### Search API

| Variable | Description |
| --- | --- |
| `SEARXNG_ENDPOINT` | SearXNG server URL with JSON format enabled |
| `SEARXNG_ENGINES` | Custom engines parameter for SearXNG |
| `SEARXNG_CATEGORIES` | Custom categories parameter for SearXNG |

### Supabase

| Variable | Description |
| --- | --- |
| `SUPABASE_ANON_TOKEN` | Supabase anonymous token |
| `SUPABASE_URL` | Supabase project URL |
| `SUPABASE_SERVICE_TOKEN` | Supabase service token |
| `TEST_API_KEY` | API key for testing with authentication enabled |

### Integrations

| Variable | Description |
| --- | --- |
| `LLAMAPARSE_API_KEY` | LlamaParse key for PDF parsing |
| `SLACK_WEBHOOK_URL` | Slack webhook URL for server health status messages |
| `POSTHOG_API_KEY` | PostHog API key for job log events |
| `POSTHOG_HOST` | PostHog host URL |

### System resources

| Variable | Description | Default |
| --- | --- | --- |
| `MAX_CPU` | Maximum CPU usage threshold (0.0-1.0). Worker rejects new jobs above this value. | `0.8` |
| `MAX_RAM` | Maximum RAM usage threshold (0.0-1.0). Worker rejects new jobs above this value. | `0.8` |

### Webhooks

| Variable | Description |
| --- | --- |
| `ALLOW_LOCAL_WEBHOOKS` | Set to `true` to allow local webhooks to your self-hosted instance |

### Auto-configured

These variables are automatically set by `docker-compose.yaml`. You should not need to change them.

| Variable | Description |
| --- | --- |
| `PLAYWRIGHT_MICROSERVICE_URL` | URL for the Playwright service |
| `REDIS_URL` | Redis connection URL |
| `REDIS_RATE_LIMIT_URL` | Redis URL for rate limiting |

## TypeScript Playwright service

By default, Firecrawl uses a Python-based Playwright service. To switch to the TypeScript version:

1. Update the `docker-compose.yml` file to change the Playwright service build path:

```yaml
# Change this:
build: apps/playwright-service

# To this:
build: apps/playwright-service-ts
```

2. Set the `PLAYWRIGHT_MICROSERVICE_URL` in your `.env` file:

```bash
PLAYWRIGHT_MICROSERVICE_URL=http://localhost:3000/scrape
```

3. Configure your proxy server in the `.env` file if needed (see the [Proxy](#proxy) section above).

## Troubleshooting

### Supabase client is not configured

**Symptom:**
```
[YYYY-MM-DDTHH:MM:SS.SSSz]ERROR - Attempted to access Supabase client when it's not configured.
[YYYY-MM-DDTHH:MM:SS.SSSz]ERROR - Error inserting scrape event: Error: Supabase client is not configured.
```

**Solution:** This error is expected when Supabase is not configured. Scraping and crawling work normally without it. Supabase configuration is not currently available for self-hosted instances.

### Authentication bypass warning

**Symptom:**
```
[YYYY-MM-DDTHH:MM:SS.SSSz]WARN - You're bypassing authentication
```

**Solution:** This warning appears when `USE_DB_AUTHENTICATION=false`. Scraping and crawling work normally. Supabase-based authentication is not currently available for self-hosted instances.

### Docker containers fail to start

**Symptom:** Docker containers exit unexpectedly or fail to start.

**Solution:**
- Check the Docker logs for error messages: `docker logs [container_name]`
- Verify all required environment variables are set correctly in the `.env` file.
- Confirm that all Docker services defined in `docker-compose.yml` are correctly configured and the necessary images are available.

### Connection issues with Redis

**Symptom:** Errors related to connecting to Redis, such as timeouts or "Connection refused".

**Solution:**
- Verify the Redis service is up and running in your Docker environment.
- Check that `REDIS_URL` and `REDIS_RATE_LIMIT_URL` in your `.env` file point to the correct Redis instance.
- Review network settings and firewall rules that may block the connection to the Redis port.

### API endpoint does not respond

**Symptom:** API requests to the Firecrawl instance timeout or return no response.

**Solution:**
- Verify the Firecrawl service is running by checking the Docker container status.
- Confirm that `PORT` and `HOST` in your `.env` file are correct and no other service is using the same port.
- Check network configuration to ensure the host is accessible from the client making the API request.

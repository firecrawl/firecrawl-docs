Firecrawl のモニタリングでは、Web サイト上のコンテンツが変更されたことを検出し、webhook またはメールで通知を受け取れます。定期的なスクレイピング、クロール、または Web 検索を実行し、各結果を最後に保持されたスナップショットと比較します。モニターを使うと、商品ページ、ドキュメント、ブログ、変更履歴、競合サイトなど、変更の追跡が重要なページを監視できます。あるいは、ゴールに一致する新しい結果を Web 全体から監視することもできます ([Web monitoring](#web-monitoring) を参照) 。

各チェックでは、ページ単位の結果が `same`、`new`、`changed`、`removed`、`error` として記録されます。監視対象の各ページの処理完了時に webhook を受け取ることも、チェックが完了するたびに webhook を受け取ることも、変更やエラーが発生した際にメールで要約を受け取ることもでき、これらを自由に組み合わせることもできます。

Firecrawl CLI からモニターを作成することもできます:

```bash CLI
firecrawl monitor create --name "Hacker News AI" \
  --schedule "every 30 minutes" \
  --goal "Alert when a new Hacker News story related to AI enters the top 10. Ignore changes to stories that are not about AI. Do not alert on changes outside the top 10." \
  --page https://news.ycombinator.com
```

<div id="goals-and-judging">
  ## ゴールと判定
</div>

意味のある変更があったときだけ通知を受けたい場合は、平易な表現で `goal` を追加します。`goal` が指定されていて `judgeEnabled` が省略されている場合、Firecrawl は自動的に判定を有効にします。判定は変更のあったページで実行され、`meaningful`、`confidence`、`reason`、`meaningfulChanges` を含む `judgment` を返します。

変更の判定はまだ行わず、ゴールだけを保存したい場合は、`judgeEnabled: false` を使用します。判定が実行されるのは、モニター に `judgeEnabled` と空でない `goal` の両方が設定されている場合だけです。

<Note>
  各 チェック では、常に元となるスクレイピングまたはクロールの料金が発生します。判定が有効な場合、judge は検証した変更済みページごとに 1 クレジット を追加で消費します。変更されたページがない チェック では、judge クレジット は消費されません。
</Note>

適切なゴールは、短く明確であることが重要です。何を alert のトリガーにするかを示し、top N、価格、役割の種類、会社、地域、トピック、status、またはエンティティなどの対象範囲も明記してください。除外条件は、意図に含まれる場合にのみ追加します。ゴールが広い場合は、その広さを保ってください。たとえば、&quot;any change&quot; の場合は、変更を見逃すようなノイズ除去の絞り込みを加えるべきではありません。

たとえば、次のような goal を持つ モニター では:

```text
AIに関連する新しいHacker Newsのストーリーがトップ10に入ったときに通知する。AIに関係のないストーリーの変更は無視する。トップ10外の変更には通知しない。
```

一致するストーリーが監視対象に入ると、このような `monitor.page` webhook が生成されることがあります:

<div id="targets">
  ## ターゲット
</div>

モニターでは、3 種類のターゲットをサポートしています。

* `scrape`: `urls` 内の各 URL に対して 1 回スクレイピングを実行します。
* `crawl`: 各チェックで `url` に対して完全なクロールを実行し、検出されたすべてのページの差分を比較します。
* `search`: 各チェックで Web 検索 `queries` を実行し、新しく一致した結果が見つかった場合にアラートを送信します。フィールドと動作については、[Web monitoring](#web-monitoring) を参照してください。

各モニターには 1～50 個のターゲットを指定できます。`retentionDays` のデフォルトは `30` で、最大 `365` まで設定できます。

ターゲットのスクレイピングのオプションは、そのまま基盤となるスクレイピング ジョブに渡されます。モニターによってトリガーされたスクレイピングでは、`maxAge` のデフォルトが `0` になるため、明示的に別の `maxAge` を設定しない限り、各チェックで毎回新しくスクレイピングが実行されます。

<div id="change-tracking">
  ## 変更追跡
</div>

デフォルトでは、Firecrawl は各ページの Markdown の差分を比較し、`same`、`changed`、`new`、`removed`、`error` のいずれかを返します。**特定の構造化フィールド** (価格、見出し、在庫フラグ、リスト内の項目など) の変更を検出したい場合は、対象の `scrapeOptions` に `modes: ["json"]` を指定した `changeTracking` フォーマットを追加して、JSON モードの変更追跡を有効にします。

<div id="email">
  ### メール
</div>

メールの要約は、チェックで変更・新規追加・削除・エラーのいずれかに該当するページがあった場合にのみ送信されます。

<EmailConfig />

モニターでゴールが設定され、判定が有効になっている場合、メールの要約では重要な変更ページが優先されます。変更されたすべてのページがノイズと判定され、新規追加・削除・エラーに該当するページもない場合、メールは送信されません。

`recipients` を省略すると、Firecrawl はシステムアラートメールの受信対象となるチームメンバーに送信します。
明示的な受信者は最大 25 件まで設定できます。

<div id="web-monitoring">
  ## ウェブモニタリング
</div>

スクレイピングモニターとクロールモニターは、指定したURLを監視します。**ウェブ**モニターはウェブ全体を対象に監視します。既知のページの差分を比較するのではなく、実行する`queries`と`goal`を指定すると、Firecrawlが各チェックでクエリを実行し、judgeがゴールに関連すると判断した**新しい**結果についてアラートを送信します。つまり、差分比較ではなく新規発見です。

各チェックでは毎回同じサイクルが実行されます。ゴールとその`queries`を受け取り、鮮度の対象期間を適用し、検索を実行し、正規URLで結果を重複排除し、任意のAI judgeにゴールに照らしてどの新規結果が意味のあるものかを判定させ、スクレイピングモニターやクロールモニターと同じwebhookおよびメールチャネルでアラートを送信します。スケジューリング、ゴール、judging、通知は、いずれも上で説明したとおりまったく同じように機能します。

<Note>
  ページモニターとクロールモニターは、指定したURL上のコンテンツの差分を比較します。ウェブモニターは、ウェブ全体から新しい結果を見つけます。スケジューリング、judge、通知の仕組みは共通です。
</Note>

<div id="search-target">
  ### 検索対象
</div>

検索対象では `type: "search"` を使用し、`urls` の代わりに、実行するクエリと結果のスコアリング方法を指定します:

```json Search target
{
  "type": "search",
  "queries": ["open source AI coding assistant launch"],
  "searchWindow": "24h",
  "maxResults": 10,
  "includeDomains": ["news.ycombinator.com"]
}
```

| Field            | Type                                             | 説明                                                                                                                                                                              |
| ---------------- | ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `type`           | `"search"`                                       | 検索対象を選択します。                                                                                                                                                                     |
| `queries`        | `string[]`                                       | 各チェックで実行する検索クエリです。                                                                                                                                                              |
| `searchWindow`   | `"5m" \| "15m" \| "1h" \| "6h" \| "24h" \| "7d"` | 新しさで絞り込むフィルターです。この期間内に公開された結果のみを対象にします。                                                                                                                                         |
| `maxResults`     | `number`                                         | 各チェックで評価する結果の総数です。`1`〜`50`。デフォルトは `10` です。これは `queries` 全体にまたがる合計上限であり (結果は最初にマージされ、重複排除されます) 、クエリごとの上限ではありません。そのため、他のクエリで先に上限に達した場合、個々のクエリから得られる結果はそれより少なくなるか、まったくなくなることがあります。 |
| `includeDomains` | `string[]`                                       | 任意。結果をこれらのドメインに限定します。`excludeDomains` とは同時に指定できません。                                                                                                                             |
| `excludeDomains` | `string[]`                                       | 任意。これらのドメインの結果を除外します。`includeDomains` とは同時に指定できません。                                                                                                                             |

<Note>
  **クレジットは `queries` の数に応じて増加します。** 各 query は最大 `maxResults` 件の結果を取得し、検索呼び出しのクレジットは、マージと重複排除の*前*に、すべての query にまたがる生の結果数に対して課金されます。したがって、query を追加すると、事前見積もりだけでなく実際の検索コストも増加します。マージ/重複排除後に Firecrawl が評価する候補は最大でも `maxResults` 件までのため、judge クレジットは選択・判定された結果数の範囲内に収まります。
</Note>

モニターレベルの `goal` と `judgeEnabled` は、[Goals and judging](#goals-and-judging) と同様に適用されます。`queries` は必須です。`goal` は judge が各新規結果をどの基準で評価するかを定めるものであり、query を生成するものではありません。

<div id="writing-good-goals-and-queries">
  ### 良いゴールとクエリの書き方
</div>

Web monitorの品質は、役割の異なる2つの要素で決まります。

* **`queries` は再現率を左右します** — つまり、各検索で何を拾うかです。狭すぎると本当に重要な出来事を取り逃がし、広すぎると判定側がノイズの絞り込みにクレジットを消費します。
* **`goal` は適合率を左右します** — つまり、取得した結果のうち実際にアラート対象になるものを決めます。判定側は新しい結果を1件ずつゴールに照らして評価するため、本当に一致しているものと、話題は近くても無関係なものを分けるのはゴールです。

この2つはどちらも調整が必要です。どれほど完璧なゴールでも、クエリで取得できなかった結果に対してはアラートできません。また、広いクエリに曖昧なゴールを組み合わせると、価値の低いアラートが絶えず発生します。

**良いクエリは、文章ではなく検索語のように書きます。**

* 文章ではなくキーワードを使います — `OpenAI new model release` とし、`tell me when OpenAI releases a new model` にはしません。
* 複数語の固有表現は引用符で囲み (`"Llama 4"`) 、同義語は `OR` でまとめます (`launch OR release OR announcement`) 。
* 各クエリは簡潔に保ちます (およそ2〜6語) 。通常は、狭いクエリをいくつも作るより、広めのクエリ1つのほうが効果的です — 余分なクエリはカバレッジを増やさないまま `maxResults` の予算を分散させてしまいます。
* *明確に異なる*対象ごとに1つのクエリを使います。1つの対象に複数の側面 (「launches、benchmarks、docs」) があっても、クエリは1つのままです。ゴールで本当に別の対象を挙げている場合 (たとえば「OpenAI、Anthropic、Google」) にだけ分けてください。
* `site:` / `-site:` 演算子はクエリに入れず、代わりに `includeDomains` / `excludeDomains` を使ってください。

**良いゴールは、一致条件を平易な言葉で示し、除外条件はそれが意図の一部である場合にだけ追加します。**

* 対象と、一致と見なす条件を明記します: *「OpenAIがまったく新しいモデルを発表したらアラートする。」*
* 多義語は明確にします: *「Firecrawl (web scraping API) 」* としておけば、判定側が消火設備の話題に引っ張られません。
* `Ignore ...` は、意図に固有の非一致だけに追加します: *「意見記事、チュートリアル、無関係なAIニュースはIgnore。」* 一般的なノイズをあらためて書く必要はありません — 判定側はすでに書式、追跡パラメータ、再インデックスによる変動を処理します。
* 意図が広いなら、そのまま広く保ちます。ゴールを絞りすぎると、本来一致すべきものまで抑えてしまいます。

**健全なmonitorの状態。** 適切に調整されたWeb monitorは、ほとんどの場合何も報告しません — 多くのcheckは `new: 0` を返します — そして、本当に新しくゴールに合致するものが現れたときだけアラートします。調整できているかどうかは、check summaryと各結果の `searchStatus` ([Statuses and dedup](#statuses-and-dedup) を参照) を見れば判断できます。

* `ignored` の結果が継続的に出るなら、クエリがノイズを拾い、それを後からゴールが却下している状態です — アラートされない結果の判定に課金され続けないよう、クエリを絞るか `searchWindow` を狭めてください。
* `watching` が頻繁に出るなら、ゴールが曖昧です — 一致条件を明確にして、判定側が判断を確定できるようにしてください。
* 活発な話題なのに結果ゼロの状態が長く続くなら、クエリが狭すぎるか `searchWindow` が短すぎます — 語を広げるか、ウィンドウを広げてください。
* ユーザーが却下するアラートが出るなら、ゴールが広すぎます — 意図に即した `Ignore ...` を追加してください。

目指すべき状態は、十分な再現率を確保しつつ高い適合率を実現することです。すべてのアラートが対応する価値を持ち、本当に重要なものを見落とさないのが理想です。

<div id="judging">
  ### 判定
</div>

各チェックが結果ごとにどの程度の処理を行うかは、モニターの `judgeEnabled` で制御されます。これは [Goals and judging](#goals-and-judging) で説明しているものと同じフラグです。判定を有効にすると、Firecrawl は一致した各結果をスクレイピングし、そのコンテンツをゴールに照らして評価します。この場合、検索呼び出しに加えて、判定した結果ごとに 1 クレジットが課金されます。`judgeEnabled: false` の場合、Webモニターは AI による判定を行わず、重複を除いた検索結果を返します。つまり、新しい SERP ヒットだけが返され、課金も検索呼び出し分のクレジットのみです (結果 10 件ごとに 2 クレジット) 。

<div id="statuses-and-dedup">
  ### ステータスと重複排除
</div>

検索結果では、スクレイピングモニターおよびクロールモニターと **同じページレベルの `status` 列挙型** を使用するため、既存の webhook や check-result の利用側はそのまま動作します。検索結果は次のいずれかに対応します。

* `new` — 初めてゴールに一致した結果です。これがアラートの対象になります。
* `same` — 以前のチェックですでに確認された結果です (新しいアラートはありません) 。
* `error` — 評価できなかった結果です (たとえば、判定用のスクレイピングがスキップされた場合) 。

より詳細な検索の判定状態は、各ページの `metadata.searchStatus` で確認でき、次のいずれかになります。

| `searchStatus` | Page `status` | 意味                                                     |
| -------------- | ------------- | ------------------------------------------------------ |
| `alert`        | `new`         | 判定で意味があるとみなされた新しい結果です — 通知を送信します。                      |
| `already_seen` | `same`        | フィンガープリントが以前のチェック結果と一致しました。                            |
| `watching`     | `same`        | まだ十分な確信が持てていない新しい結果です。追跡はされますが、アラートは送信されません。           |
| `ignored`      | `same`        | ゴールに対して意味がないと判定された新しい結果です。                             |
| `skipped`      | `error`       | このチェックでは結果を判定できませんでした (たとえば、スクレイピングの失敗や判定精度の低下が原因です) 。 |

結果がアラートになるのは、最初に `new` として現れたときの一度だけです。重複排除のキーは正規 URL のみです (title と snippet は意図的にフィンガープリントに含めていないため、title/snippet が変わっても再通知されません) 。キーが URL であるため、現実世界では 1 つの出来事でも、それが複数の記事 URL で報じられている場合は、出来事ごとに 1 回ではなく、URL ごとに 1 回アラートされます。

モニターの `goal` または `queries` を編集すると、`goalVersion` が更新され、以前の判定結果は無効になります。再評価は一括再判定ではなく、必要になった時点で遅延して行われます。つまり、既存の結果がまとめて再判定されることはありません。代わりに、各結果は次回のチェックで再び現れたときに再判定され、その時点で新しい `goalVersion` が適用されます。再び現れない結果は、再出現するまで古い判定結果と `goalVersion` を保持します。

<div id="create-a-web-monitor">
  ### Webモニターを作成する
</div>

Webモニターはスクレイピングモニターと同じ方法で作成します。違うのは、対象が (`queries`、`searchWindow`、`maxResults`、および任意のドメインフィルターを指定した `type: "search"`) であり、URL を指定しない点だけです。

<CodeGroup>
  ```python Python
  from firecrawl import Firecrawl

  firecrawl = Firecrawl(
    # Monitor エンドポイントには API キーが必要です:
    api_key="fc-YOUR-API-KEY",
  )

  monitor = firecrawl.create_monitor(
      name="AI coding assistant launches",
      schedule={"text": "every 30 minutes", "timezone": "UTC"},
      goal="Alert when a new open-source AI coding assistant is announced. Ignore funding rounds and unrelated AI news.",
      targets=[
          {
              "type": "search",
              "queries": ["open source AI coding assistant launch"],
              "searchWindow": "24h",
              "maxResults": 10,
          }
      ],
      notification={
          "email": {
              "enabled": True,
              "recipients": ["alerts@example.com"],
              "includeDiffs": True,
          }
      },
  )

  print(monitor.id)
  ```

<div id="pricing">
  ## 料金
</div>

モニター にモニターごとの個別料金はありません。各チェックでは、実行されるスクレイピング、クロール、または検索の分のクレジットが消費され、さらに有意な変更の判定を有効にしている場合は、判定で有意な変更ありと認められたページごとに追加で 1 クレジットが消費されます。

| コンポーネント                                          | クレジット                                              |
| ------------------------------------------------ | -------------------------------------------------- |
| スクレイピングモニター                                      | URL ごとに、チェック 1 回あたり 1 クレジット                        |
| クロールモニター                                         | 検出されたページごとに、チェック 1 回あたり 1 クレジット                    |
| ウェブモニター                                          | 10 件の結果ごとに、チェック 1 回あたり 2 クレジット                     |
| ウェブモニターの判定                                       | AI 判定が有効な場合、判定された結果ごとに 1 クレジット (結果のスクレイピングと評価を含む)  |
| 有意な変更の判定が有効                                      | 判定で変更ありと認められたページごとに追加で 1 クレジット                     |
| Format アドオン (JSON、PDF、question、enhanced mode など) | 単体の[scrape](/ja/features/scrape)と同じ                   |
